# 项目文档体系

本参考定义默认文档的职责、边界与建议结构。根据项目规模压缩章节，但不要改变每份文档的核心职责。

## 目录

- 基本原则
- `AGENTS.md`
- `docs/VISION.md`
- `docs/ARCHITECTURE.md`
- `docs/ABSTRACTIONS.md`
- `docs/GETTING-STARTED.md`
- `docs/adr/README.md`
- 兼容入口

## 基本原则

- `AGENTS.md` 保存可执行的工作规则，不承担完整架构说明。
- `VISION.md` 解释为什么做、为谁做和明确不做什么。
- `ARCHITECTURE.md` 描述系统如何组成和运行。
- `ABSTRACTIONS.md` 描述项目中的核心概念、模型和不变量。
- `GETTING-STARTED.md` 帮助新贡献者运行项目并定位代码。
- `docs/adr/` 保存重大决策形成时的背景和理由。
- 当前事实与计划设计分开表述。不要用将来设计填补当前实现的空白。
- 同一事实只维护一次。选择权威文档后，从其他文档链接过去。

## `AGENTS.md`

### 职责

作为项目级 coding agent 的规范入口，写入具体、可执行、可验证的规则。

### 默认结构

1. 项目一句话说明。
2. 开始任务前必须阅读的文档。
3. 开发流程和分支、提交约定。
4. 编码和项目专用规则。
5. 构建、测试、静态检查和验收命令。
6. 安全、数据和生成文件限制。
7. 文档维护触发条件。
8. 平台或工具的关键注意事项。

只写目标项目真实采用的规则。不要凭空加入 TDD、覆盖率阈值、发布流程、外部服务或强制提交策略。

### 文档维护映射

| 变更类型 | 应检查的文档 |
| --- | --- |
| 产品目标、受众或非目标变化 | `docs/VISION.md` |
| 模块边界、数据流、依赖或部署变化 | `docs/ARCHITECTURE.md`，必要时新增 ADR |
| 领域模型、公共类型、状态或命名约定变化 | `docs/ABSTRACTIONS.md`，必要时新增 ADR |
| 环境要求、命令、目录或常见操作变化 | `docs/GETTING-STARTED.md` |
| 新的长期架构选择及其取舍 | `docs/adr/`，并同步当前状态文档 |

## `docs/VISION.md`

### 职责

为实现取舍提供产品方向，不记录代码细节。

### 默认结构

1. 项目概述。
2. 要解决的问题。
3. 目标用户或使用者。
4. 项目目标与成功表现。
5. 非目标和边界。
6. 核心原则。
7. 当前阶段或近期方向。

如果仓库无法证明愿景内容，向用户询问。不要把脚手架介绍、依赖描述或 agent 的推测改写成产品愿景。

## `docs/ARCHITECTURE.md`

### 职责

描述当前系统结构、边界和运行方式，并将未实现设计清楚标为计划。

### 默认结构

1. 当前状态摘要。
2. 架构原则和不变量。
3. 技术栈及每项技术的职责。
4. 系统上下文和外部依赖。
5. 主要模块与所有权边界。
6. 关键数据流或请求流程。
7. 数据存储、缓存和一致性规则。
8. 构建、部署与运行环境。
9. 安全、可靠性和可观测性边界。
10. 相关 ADR 和关键源码入口。

复杂关系优先用 Mermaid 图表达。图必须与文字一致，并使用真实模块名。

## `docs/ABSTRACTIONS.md`

### 职责

记录跨文件、跨模块或跨语言理解代码时必须掌握的项目概念。

### 默认结构

1. 领域术语表。
2. 核心实体、值对象或公共类型。
3. 标识、命名和序列化约定。
4. 生命周期与状态转换。
5. 所有权和数据权威来源。
6. 跨层映射与边界转换。
7. 关键不变量和兼容性约束。
8. 典型使用示例及对应源码位置。

简单项目可以保持精简，但仍应解释项目特有概念。不要复制完整 API 或逐文件列出内部类型。

## `docs/GETTING-STARTED.md`

### 职责

让第一次接触项目的人能够准备环境、运行验证并找到修改入口。

### 默认结构

1. 前置环境和支持平台。
2. 安装、配置和环境变量。
3. 最短可运行步骤。
4. 构建、测试、检查和调试命令。
5. 目录结构。
6. 关键文件索引。
7. 常见开发任务。
8. 常见故障及恢复方法。

命令必须来自脚本、CI 或用户确认。计划中的命令不得写成可执行的当前命令。

## `docs/adr/README.md`

### 职责

解释何时创建 ADR、文件命名、状态变化和内容格式。默认在 README 内给出模板，不额外创建空白 ADR 文件。

### 文件和状态规则

- 文件名使用 `NNNN-short-title.md`，编号单调递增。
- 状态使用 `proposed`、`active`、`superseded` 或 `retired`。
- 一份 ADR 只记录一个长期决策。
- ADR 生效后保留历史；改变决策时创建新 ADR，并建立替代关系。
- 修复、局部样式调整和没有长期取舍的重构通常不需要 ADR。

### ADR 模板结构

```text
# NNNN — 决策标题

- Status: proposed
- Date: YYYY-MM-DD
- Supersedes: none

## Context

促成决策的背景、约束和作用力。

## Decision

明确写出选择。

## Options considered

列出主要候选方案及取舍。

## Consequences

记录正面影响、代价和重新评估条件。
```

文档采用中文时，可以将章节标题翻译为“背景”“决策”“考虑的方案”“影响”，但同一项目内保持一致。

### 固定脚本

使用技能 `scripts/` 中的固定入口，不要在每次任务中重新设计 ADR 脚本：

- Windows PowerShell 使用 `python` 直接运行 `create_adr.py` 与 `validate_agent_docs.py`。
- Bash 使用 `python3` 直接运行相同脚本。
- 两种终端共用同一实现，确保编号、模板和校验结果一致。

创建脚本自动选择下一个四位编号，拒绝覆盖文件，并要求提供背景、决策、至少一个候选方案和至少一个影响。验证脚本检查核心文档、UTF-8、占位符、本地链接和 ADR 结构。脚本的完整调用格式以 `SKILL.md` 为准。

## 兼容入口

仅在需要时创建 `CLAUDE.md` 或 `GEMINI.md`。文件应保持简短并明确要求读取根目录 `AGENTS.md`。如果既有兼容文件包含独有且仍有效的规则，先将共享规则合并到 `AGENTS.md`；无法安全合并时保留原内容并添加入口，不得直接替换。
