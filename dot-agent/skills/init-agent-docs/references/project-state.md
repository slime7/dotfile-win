# 项目状态判断

先判断项目状态，再选择创建或合并流程。状态反映项目特有实现的成熟度，而不是目录大小或初始化时间。

## 三种状态

### 空白新项目

常见证据：

- 目录为空，或只有版本控制、编辑器配置和简短说明。
- 技术栈尚未确定，或只有用户明确描述的计划。
- 没有可运行的项目特有功能。

处理方式：先提取用户提供的项目信息，再询问无法发现的愿景与约束。架构内容必须区分计划和当前状态。

### 脚手架新项目

常见证据：

- 存在依赖清单、锁文件、源码、测试、构建配置和 Git 提交。
- README 仍是生成器的通用说明，或仍包含默认项目名和启动提示。
- 页面、接口、组件或测试仍是计数器、欢迎页、示例路由等默认内容。
- 目录结构主要反映框架约定，没有项目特有领域模块。
- 尚无生产部署、迁移、真实外部集成或稳定公共接口。

处理方式：按新项目初始化文档。将脚手架提供的运行能力描述为开发基础，不得描述为产品能力。保留已经验证的少量定制内容。

### 已有项目

常见证据：

- 存在项目特有的业务流程、领域模型、公共接口或持久化结构。
- 测试覆盖真实行为，而非生成器示例。
- 存在真实部署、迁移、发布、监控或外部集成配置。
- Git 历史反映持续开发和架构演进。
- 已有长期维护的架构、贡献、运维或决策文档。

处理方式：审计后合并。代码与配置用于验证当前事实，现有文档用于理解规则和原因，两者冲突时不要静默选择一方。

## 判断顺序

1. 查找生成器指纹和默认示例。
2. 查找项目特有名称、领域概念和业务入口。
3. 检查测试是在验证示例还是实际行为。
4. 检查部署、迁移、集成和发布是否真实使用。
5. 抽查 Git 提交是否只有初始化，或已包含持续演进。
6. 综合证据分类，不使用单一指标。

## 边界情况

### 大型脚手架

框架可能生成数百个文件和完整测试配置。只要主体仍是通用模板且没有项目特有实现，就属于脚手架新项目。

### 已做少量定制

如果只有名称、主题、几个页面或配置被替换，而核心能力尚未形成，仍归为脚手架新项目。文档中保留已验证的定制，不把默认示例计入项目功能。

### 小型但已投入使用

文件很少不等于新项目。只要已有稳定用途、真实用户流程、发布方式或项目特有约束，就按已有项目处理。

### 代码与用户描述不一致

代码可能落后于计划，用户描述也可能落后于实现。将代码可验证内容列为当前实现，将用户明确确认但未实现的内容列为计划；如果两者直接冲突，先询问用户。

## 证据记录

写文档前建立简短事实清单，每项至少记录：

| 内容 | 状态 | 证据 |
| --- | --- | --- |
| 项目名称 | 当前实现或用户确认 | 清单、README 或用户输入 |
| 运行命令 | 当前实现 | 项目脚本或 CI |
| 产品目标 | 用户确认或未知 | 用户输入或权威愿景文档 |
| 架构关系 | 当前实现或计划 | 源码、配置或用户确认 |

事实清单用于写作判断，无需默认写入目标仓库。
