# 已有项目的审计与合并

已有项目的目标是建立一致的权威文档体系，而不是用统一模板覆盖已有知识。

## 审计步骤

1. 列出所有 agent 指令、README、贡献指南、架构文档、设计文档和 ADR。
2. 搜索构建、测试、发布、目录、架构和编码规则在多处的重复描述。
3. 将每条内容标为：已验证、过时、冲突、重复、未知来源或计划。
4. 为每类事实选定权威文档。
5. 先保留和迁移独有知识，再删除重复；无法确认的内容保持原位并向用户说明。
6. 记录实际文档路径映射，后续创建、链接和校验都使用该映射。

## 权威位置

| 内容 | 默认权威位置 |
| --- | --- |
| agent 必须执行的流程与约束 | `AGENTS.md` |
| 产品目标、受众和边界 | `docs/VISION.md` |
| 系统组成、数据流和部署 | `docs/ARCHITECTURE.md` |
| 领域模型、公共概念和不变量 | `docs/ABSTRACTIONS.md` |
| 环境、命令、目录和常见任务 | `docs/GETTING-STARTED.md` |
| 长期技术选择的背景与取舍 | `docs/adr/` |

项目已有更合适的权威文档时，优先保留其位置，并从其他入口链接过去。不要为了统一文件名复制大段内容，也不要同时创建默认位置的副本。调用校验脚本时使用路径参数描述实际位置。

## 合并规则

- 逐段合并，不整体重写已有文件。
- 保留项目专用规则、人工解释、事故经验和无法从代码推导的原因。
- 用代码、配置和 CI 验证命令、路径和当前行为。
- 过时内容只有在新事实有明确证据时才更新或删除。
- 冲突内容不静默覆盖。记录冲突双方和证据，请用户确认会影响行为的选择。
- 移动内容后在原位置留下清晰链接，除非原位置只是无引用的重复内容。
- 兼容入口中的共享规则先迁入本次采用的实际 agent 入口，再将兼容文件缩减为入口。

## ADR 处理

- 默认创建或补全 `docs/adr/README.md`，不根据当前代码反推历史理由。
- 只有现有设计文档、提交说明或用户确认同时证明“选择了什么”和“为什么选择”，才补写历史 ADR。
- 已有 ADR 使用不同编号或状态体系时，保持现有体系并在 README 中补充说明，不批量改名。
- 当前架构写入 `ARCHITECTURE.md`；ADR 记录形成选择时的背景，两者不能互相替代。

## 完成检查

- 原有独有内容仍可找到。
- 命令和路径与仓库一致。
- 没有两份互相矛盾的共享 agent 规则。
- 默认文档之间通过相对链接连接，且没有大段重复。
- 不确定内容被明确标注或列入完成报告。
