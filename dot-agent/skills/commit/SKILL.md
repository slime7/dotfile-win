# 提交更改

帮助用户提交代码更改，根据 diff 生成规范的提交信息，并遵循仓库已有的约定。

## 准则

- **未经询问，禁止修改已有提交**。
- **未经用户明确批准，禁止强制推送或推送**。
- **禁止跳过 pre-commit 钩子**（不使用 `--no-verify`）。
- **禁止跳过签名提交**（不使用 `--no-gpg-sign`）。
- **除非用户明确要求，禁止回退、重置或丢弃用户的更改**。
- 检查是否存在不应被提交的明显密钥或生成产物。如存在风险——询问用户。
- 当对暂存、约定或信息内容有疑问时——询问用户。

## 工作流

### 1. 了解仓库的提交约定

运行以下命令采样最近的提交和用户自己的提交：

```
# 最近的仓库提交（了解整体风格）
git log --oneline -20

# 用户最近的提交（了解个人风格）
git log --oneline --author="$(git config user.name)" -10
```

分析输出，按以下优先级确定提交信息的格式约定：

1. **优先沿用仓库和用户的既有风格** — 如果日志中存在清晰、统一的提交约定（如 Conventional Commits、Gitmoji、ticket 前缀等），则所有生成的提交信息**必须**遵循该约定。
2. **回退规则** — 如果日志记录过少（< 5 条）或风格不统一、无法识别出明确的约定，则使用以下默认格式：

   ```
   <type>(<scope>): <subject>
   ```

   - `<type>`：变更类型（如 `feat`、`fix`、`docs`、`style`、`refactor`、`perf`、`test`、`build`、`ci`、`chore`、`revert`）。
   - `<scope>`：可选，受影响的模块或范围。
   - `<subject>`：简要描述变更内容。

无论使用哪种约定，`<subject>`（或主题行的描述部分）**所使用的语言必须与用户输入的语言一致**。例如用户用中文交流则使用中文，用户用英文交流则使用英文。

### 2. 检查仓库状态

```
git status --short
```

- 如果**没有更改**（工作区干净，无暂存），通知用户并停止。
- 如果有**已暂存的更改**，直接使用这些更改，不暂存任何未暂存的更改。
- 如果**只有未暂存的更改**，暂存所有内容（`git add -A`），然后继续。

### 3. 生成提交信息

获取将要提交的完整 diff：

```bash
git diff --cached --stat
git diff --cached
```

根据 diff 和步骤 1 确定的约定，起草提交信息：

- **主题行**（≤ 72 字符），遵循步骤 1 确定的格式约定，概括变更内容。
- 主题行描述部分的语言与用户输入保持一致。
- 可选的**正文**，仅在 diff 较复杂时解释*为什么*进行此更改。
- 当分支名或相关上下文中包含 issue/ticket 编号时进行引用。
- 聚焦于变更的意图，而非逐文件罗列。

### 4. 提交

构造 `git commit` 命令并执行：

```
git commit -m "<subject>" -m "<body>"
```

### 5. 确认

提交完成后：

- 运行 `git status --short` 确认提交已完成。
- 运行 `git log --oneline -1` 显示新提交。
- 如果 pre-commit 钩子修改了文件或阻止了提交，准确说明发生了什么。
- 如果钩子在提交尝试后重写了文件，不要自动修改。告知用户发生了什么变化，并询问是否需要暂存并提交这些后续编辑。
